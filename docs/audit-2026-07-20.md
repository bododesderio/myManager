# MyManager — Full Engineering Audit
**Date:** 2026-07-20
**Scope:** 861 files, ~60k LOC TS/TSX. API (43 NestJS modules), Web (88 pages), Mobile (31 screens), 91 Prisma models.
**Method:** 6 parallel deep audits (architecture, security, performance, frontend, devops, correctness). Every finding below was re-verified by hand against source. Findings the audits got *wrong* are listed in Appendix A so they don't get re-litigated later.

---

> **Note on disclosure.** This repository is public. Concrete reproduction steps
> for the Critical findings have been deliberately withheld from this document.
> Severity, affected file:line, and the applied fix are retained so a reviewer can
> verify remediation. The unredacted audit is held privately by the maintainer.


## 0. Verdict

**Not production-ready. Do not accept real payments or real customer data in the current state.**

Not because the codebase is bad — it isn't. The scope is genuinely impressive, the layering is clean, the platform integrations are real, and several security primitives are implemented with real care. The blocker is narrow and specific:

**Three confirmed critical defects, all in authorization and billing, all trivially exploitable by any authenticated user.** Each is a small, well-understood fix. This is roughly 2-3 days of work, not a rewrite.

The second-order problem is more important than any single bug: **the readme claims a security control that does not exist** (PostgreSQL RLS). That claim is why the broken tenancy guard below has no backstop. Documentation that overstates controls is a security defect in itself, because it stops people from looking.

### Context that frames everything

`git log` shows **32 commits, one author, 2026-03-18 → 2026-04-13** — ~60k LOC in about three weeks, then three months dormant. Nothing here has been exercised by production traffic. "Production-ready" in the readme is a self-assessment written by the same process that wrote the code, in the same window. Treat every "Working" label as "the code path exists," not "this has published a real post."

The commit sequence on 2026-04-06 (`fix(critical):` → `fix(high):` → `fix(medium):` → `fix(low):`) shows a prior audit was worked through by severity. Some of what follows may be known and deferred.

## 1. Current Architecture

### 1.1 Topology

```
Turborepo (pnpm workspaces)
├── apps/api        NestJS 10 + Prisma 6 + BullMQ      → Railway
├── apps/web        Next.js 15 App Router + React 19   → Vercel
├── apps/mobile     Expo SDK 53                        → EAS
├── apps/extension  browser extension (minimal)
└── packages/       config, constants, emails, seo, translations, types, utils, validators
                    (note: NO packages/ui — see §5.2)

Runtime: PostgreSQL 16 · Redis 7 (BullMQ + locks) · Cloudflare R2 · Resend · Flutterwave · Anthropic + Replicate
```

### 1.2 Request path

```
Client → Next.js (SSR + /api proxy routes)
       → NestJS
         → APP_GUARD chain: JwtAuthGuard → RolesGuard → CsrfGuard → WorkspaceMemberGuard → ThrottlerGuard
         → Controller → Service → Repository → Prisma
         → (async) BullMQ queue → Worker
```

Global guard registration is at `apps/api/src/app.module.ts:137-143`. **This is the single best architectural decision in the codebase** — it is fail-closed by default, so there are no forgotten unprotected controllers. The vulnerabilities in §2 are logic bugs *inside* the guards, not coverage gaps.

### 1.3 Publishing pipeline (the core product loop)

```
POST /posts                  → PostsService.create()        status=DRAFT
POST /posts/:id/publish      → status=QUEUED, enqueue per platform
   ─ or ─
ScheduledPostsCron (*/1min)  → withDistributedLock('scheduled-posts', 55s)
                             → find SCHEDULED where scheduled_at <= now
                             → status=QUEUED, enqueue per platform

BullMQ: 10 queues (publishing-facebook, -instagram, -x, ...) attempts:5, exponential backoff 60s
   → BasePublishingWorker.process()
     → load post + media, load socialAccount, decryptToken()
     → upsert PostPlatformResult status=PUBLISHING
     → buildPayload() [subclass] → publish() [subclass] → real platform HTTP API
     → update PostPlatformResult status=PUBLISHED
     → checkAllPlatformsPublished() → Post.status = PUBLISHED | PARTIALLY_PUBLISHED
     → auditLog.create()
```

`BasePublishingWorker` + 10 thin subclasses is the right shape. Platform-specific logic is properly isolated; shared orchestration lives in the base. This is well done.

### 1.4 What is genuinely good

Stated plainly, because a fair audit records what works:

| Area | Evidence |
|---|---|
| **Fail-closed global guards** | `app.module.ts:137-143` — 5 guards, no per-controller opt-in required |

## 2. CRITICAL — fix before any real traffic

### C1. Payment bypass: the client chooses which plan they receive
**`apps/api/src/modules/billing/billing.service.ts:196`** · CONFIRMED

```ts
const planSlug = data.plan || verified.meta?.plan;   // client input takes PRIORITY
...
billingCycle: data.billing_cycle || verified.meta?.billing_cycle || 'monthly',   // :216 same flaw
...
amount: Number(verified.amount ?? 0),                // :223 amount is RECORDED, never COMPARED
```

`verifyPayment` calls Flutterwave, confirms the transaction is `successful`, and then grants whatever plan **the client asked for in the request body**. The verified amount is written to `BillingHistory` but never checked against `plan.price_monthly_usd`.

**Exploit:** *Reproduction steps withheld from this public document.* The full walkthrough is held privately by the maintainer. Severity, location and remediation are recorded below, which is what a reviewer needs to verify the fix.

**Fix:** Derive `plan` and `billing_cycle` **exclusively** from `verified.meta` (server-controlled, set at checkout initiation). Then assert the amount matches before activating:

```ts
const planSlug = verified.meta?.plan;          // never data.plan
const cycle    = verified.meta?.billing_cycle ?? 'monthly';
const plan     = await this.repository.findPlanBySlug(planSlug);

const expected = cycle === 'annual' ? plan.price_annual_usd : plan.price_monthly_usd;
const paid     = new Prisma.Decimal(verified.amount ?? 0);
if (!paid.equals(expected) || verified.currency !== 'USD') {
  throw new BadRequestException('Payment amount does not match plan price');
}
```
(Currency needs the FX-conversion path handled too — see §3.4.)

### C2. Cross-tenant read/write: the tenancy guard is fail-open and trusts attacker input
**`apps/api/src/common/guards/workspace-member.guard.ts:20-21, 44-54`** · CONFIRMED

Two independent bypasses in the one guard protecting all multi-tenant data:

```ts
const workspaceId = await this.resolveWorkspaceId(request);
if (!workspaceId) return true;                    // :21  UNRESOLVED → ALLOW
```
```ts
const explicitWorkspaceId = this.getString(
  request.params?.workspaceId ?? request.query?.workspaceId ?? request.body?.workspaceId,
);
if (explicitWorkspaceId) return explicitWorkspaceId;   // :51  attacker input beats resource ownership
return this.inferWorkspaceIdFromResource(request);
```

**Bypass A — explicit-wins.** A caller-supplied workspace identifier is trusted ahead of the resource's actual owner, so the membership check can be satisfied against a workspace the caller legitimately belongs to while the resource belongs to another. The service then reads by primary key with no workspace scope (`posts.repository.ts:38-40`). This defeats all 10 modules the inference list exists to protect. *Concrete request shape withheld.*

**Bypass B — fail-open.** `inferWorkspaceIdFromResource` handles only 10 route prefixes. Modules with no branch return `null` → line 21 allows unconditionally. Confirmed unscoped: `campaigns`, `bio-pages`, `templates`, `rss`, `competitors`, `listening`. Also `billing.service.ts:163` — `getInvoice(id)` takes no `userId`; any user reads any invoice.

Worst instance: `templates.service.ts:37` `createPostFromTemplate` copies the *stolen* template's `workspace_id` — escalating cross-tenant **read** into cross-tenant **write**.

**Fix (layered — do all three):**
1. Delete the explicit-`workspaceId` branch. Always infer from the resource.
2. Flip `:21` to deny-by-default (`throw new ForbiddenException()`).
3. **The durable fix:** push `workspace_id` into every repository `where` clause. A guard with a hand-maintained route-prefix list *will* rot on the next module added. The database should enforce tenancy, not a lookup table someone has to remember to update.

### C3. `@WorkspaceRoles()` is a decorative no-op — role checks are not enforced
**`apps/api/src/common/guards/workspace-role.guard.ts:12`** · CONFIRMED

`grep -rn WorkspaceRoleGuard` returns **exactly one hit — its own definition.** It is not in the `APP_GUARD` list (`app.module.ts:137-143`) and appears in no `@UseGuards`. The decorator is applied in controllers and enforces nothing.

Consequently `workspaces.service.ts:48-77` has **zero authorization**:
```ts
async listMembers(workspaceId)                    { return this.repository.findMembers(workspaceId); }
async updateMemberRole(workspaceId, memberId, role) { ... }
async removeMember(workspaceId, memberId)         { ... }
async updateApprovalConfig(workspaceId, data)     { ... }
```
(`inviteMember` is the only method calling `ensureAdminAccess`.)

**Exploit:** *Reproduction steps withheld.* Combined with C2, the member-management endpoints were reachable across tenant boundaries, exposing roster data (names + emails) and permitting role changes, member removal and approval-policy changes in another tenant. Severity and remediation below.

**Fix:** Register `WorkspaceRoleGuard` as `APP_GUARD` **after** `WorkspaceMemberGuard` (it depends on `request.workspaceMember` being populated). Add explicit ownership assertions in the service regardless — a member roster is too valuable to protect with a guard alone.

### C4. The claimed RLS backstop does not exist
**`apps/api/src/common/utils/rls.ts`** · CONFIRMED

The readme claims "PostgreSQL Row-Level Security on 31 tables." Evidence:
- `grep -rn "CREATE POLICY\|ROW LEVEL SECURITY"` across all `.sql` and `.ts` → **0 results repo-wide.**
- `setWorkspaceContext()` has **zero call sites** outside its own definition and a doc comment. No Prisma middleware wires it.
- Even if policies existed: `docker-compose.yml:41` connects as `mymanager`, the schema-owning role. **Table owners bypass RLS** unless `FORCE ROW LEVEL SECURITY` is set.

This is why C2 has no second line of defense. The UUID validation in `rls.ts:24-30` *is* sound and does prevent injection into the `$executeRawUnsafe` on line 31 — but the whole function is dead code.

**Fix:** Either implement it properly (policies + `FORCE` + a **non-owner** application role + a Prisma `$extends` setting context per request), or **delete the claim from the readme today**. The current state is worse than either option, because it advertises a control that isn't there.

## 3. HIGH

### H1. Duplicate posts to real social accounts
**`workers/platforms/base.worker.ts:58-62`** · CONFIRMED

The `upsert` sets `PUBLISHING` unconditionally — no guard on an existing `PUBLISHED` state. Jobs are enqueued `attempts: 5`. If `publish()` succeeds but the process dies before the DB write at `:69`, BullMQ retries and **posts again** — up to 5 duplicates on a real customer account. Spam-detection risk on top of credibility loss.

**Fix:** guard at entry, and make the platform call idempotent where the API supports it.
```ts
const existing = await this.prisma.postPlatformResult.findUnique({
  where: { post_id_platform: { post_id: postId, platform } },
});
if (existing?.status === 'PUBLISHED') {
  this.logger.warn(`Post ${postId}/${platform} already published — skipping retry`);
  return;
}
```
The `@@unique([post_id, platform])` constraint already in the schema is the right primitive; it just isn't being read before acting.

### H2. Scheduled posts can silently vanish
**`crons/scheduled-posts.cron.ts:36-38`** · CONFIRMED (found during verification; not in any audit report)

```ts
await this.prisma.post.update({ where: { id: post.id }, data: { status: 'QUEUED' } });
for (const platform of post.platforms) { /* ...enqueue... */ }
```

Status flips to `QUEUED` **before** the platforms are enqueued, non-atomically. If the process dies mid-loop, the post sits at `QUEUED` with only some platforms dispatched — and the cron only ever selects `status: 'SCHEDULED'`. **It will never be retried and never errors.** For a scheduling product, a post that disappears without failing is worse than one that fails loudly.

**Fix:** enqueue first, then flip status; or record per-platform dispatch state and add a reconciliation sweep for posts stuck in `QUEUED` beyond a threshold.

### H3. Money loses precision at the exact moment it becomes a charge
**`prisma/schema.prisma:294-296, 320, 344, 357`** · CONFIRMED

```prisma
price_monthly_usd Decimal @db.Decimal(10,2)   // :294  CORRECT
price_annual_usd  Decimal @db.Decimal(10,2)   // :295  CORRECT
seat_price_usd    Float                        // :296  WRONG
billing_amount    Float                        // :320  WRONG  (Subscription)
unit_price        Float                        // :344  WRONG  (SubscriptionItem)
amount            Float                        // :357  WRONG  (BillingHistory)
```

Money is **Decimal in the price catalog and Float in the ledger** — more dangerous than uniform Float, because it's correct everywhere someone would spot-check. `billing.service.ts:64` does `Number(plan.price_monthly_usd)`, destroying the exact value the moment it becomes a real charge. `:68`'s `Number((amount * rate).toFixed(2))` is a band-aid over the same wound.

**Fix:** migrate the four Float columns to `Decimal @db.Decimal(10,2)`; keep values as `Prisma.Decimal` end-to-end and never round-trip through JS `number`. Backfill is lossy-but-stable since existing values were written from 2dp sources.

### H4. Every logout full-scans the sessions table
**`prisma/schema.prisma:202-212`** · CONFIRMED

`Session.user_id` has **no index**. And this table does double duty — `auth.repository.ts` stores *refresh tokens* in `Session`, so `deleteAllRefreshTokensForUser()` runs `deleteMany({ where: { user_id } })`: a sequential scan on the hottest security path. Every logout, password change, and "sign out all devices" scans the whole table.

**Fix:** `@@index([user_id])`. One line, large payoff.

**Related — architectural smell:** a well-designed `RefreshToken` model exists (`:239-253`) with `used_at`, `revoked_at`, and correct indexes — textbook reuse detection — but appears **unused**; the live path uses `Session`. Either a stalled migration or a rewrite that abandoned the better design. Resolve the direction before building on either.

### H5. Puppeteer launches a full Chromium per PDF
**`workers/report-generator.worker.ts:69-86`** · CONFIRMED

`puppeteer.launch()` per report: ~150-200MB RSS and 3-5s startup each. 10 concurrent reports ≈ 2GB of browser processes. Also `:106-116` buffers an entire unbounded `findMany` of analytics into memory for CSV.

**Fix:** a `generic-pool` of 2-3 reusable browsers, `concurrency` capped to pool size, and stream CSV in 1k-row chunks instead of buffering.

### H6. Six independent AES-256-GCM implementations
CONFIRMED — worse than the architecture audit's claim of three:
```
workers/token-refresh.worker.ts:96,112
workers/analytics-sync.worker.ts:231
workers/platforms/base.worker.ts:121
modules/system-config/system-config.service.ts:66,82
modules/auth/auth.service.ts:501,517            ← TOTP secrets
modules/social-accounts/social-accounts.service.ts:278,294
```
Plus `packages/utils/encryption.ts:14` uses a **16-byte IV** where the others correctly use 12 — already divergent.

This spans two trust domains (OAuth tokens *and* 2FA secrets). A crypto fix applied to one file stays broken in five. **Highest-value refactor in the codebase** — not for elegance, for blast radius.

**Fix:** one injectable `CryptoService` (`encrypt`/`decrypt`, key loaded and length-validated once at boot), delete the other six.

### H7. Flutterwave webhook proves nothing about the payload
**`billing.controller.ts:116-131`** · CONFIRMED

`verif-hash` is a **static shared secret**, not an HMAC over the body — identical on every request, so it authenticates nothing about *this* payload. No timestamp, no nonce. `handleChargeCompleted:364` dedupes on `flw_ref`, which stops exact replays but not forged bodies.

This is a Flutterwave protocol limitation, so **compensate**: never trust webhook body values. On every `charge.completed`, re-fetch `GET /transactions/:id/verify` server-side and reconcile amount + currency + plan before mutating a subscription. (Your own internal forwarder at `webhooks.controller.ts:69-92` already does signing correctly — follow that model where you control both ends.)

### H8. Registration is not atomic
**`modules/auth/auth.service.ts:72-93`** · CONFIRMED

Five sequential writes — user → workspace → member → preferences → plan — with no transaction. A failure at step 3 leaves a user who owns nothing and cannot access any workspace: a permanently broken account that also blocks re-registration via the unique-email check at `:66`.

The whole API has exactly **two** meaningful write transactions (`auth.repository.ts:106`, `data-deletion.cron.ts:24`); the two in `blog.service.ts:59,135` are `$transaction([...])` *array* form — batched reads, not write atomicity.

**Fix:** wrap registration and `verifyPayment`'s activate-and-record sequence in `prisma.$transaction`.

### H9. Three critical dependency CVEs
`pnpm audit`: **203 vulnerabilities, 3 critical, 91 high.** Criticals: `handlebars <=4.7.8` (JS injection via AST type confusion — **you render email templates**), `protobufjs <7.5.5` (RCE), `shell-quote <=1.8.3`. Three months dormant means this has only drifted further. `handlebars` first.

## 5. Frontend

### 5.1 Numbers
88 pages · 114 `'use client'` (~76%) · 28 components · **0 `loading.tsx`** · 39 `error.tsx` (44% coverage) · 73 raw `fetch()` calls bypassing the API client · **83 `any` in `app/`, 11 in `lib/`**

### 5.2 The structural problems

**No `packages/ui`.** Despite nine shared packages, there is no shared component library. 28 components live in `apps/web/components/`, hand-rolled on raw `<button>`/`<input>` with Tailwind. Across 49 dashboard pages this guarantees drift. (Stack defaults call for shadcn/ui; it isn't present.)

**`packages/validators` has zero imports in the web app.** Zod schemas exist for user/post/brand/plan, and every form re-implements validation by hand in `useState`. Shared schemas nobody shares are *worse than none* — they read as coverage. `LoginForm` alone carries 8 `useState` hooks doing what one resolver would.

**73 raw `fetch()` calls** bypass `lib/api/client.ts` and therefore its auth-refresh and CSRF interceptors — including `LoginForm.tsx:46` and `SignupForm.tsx:51`.

**Zero `loading.tsx`** means no Suspense boundaries anywhere; skeletons exist (`components/skeletons/`) but aren't wired to route transitions. 59% of pages have no `error.tsx`, so an API failure white-screens to the root boundary.

**Accessibility is not AA.** Icon-only buttons without `aria-label` (e.g. password toggle `LoginForm.tsx:167-185`), no focus management in modals, no visible focus-ring strategy.

### 5.3 Mobile
~24/31 screens wired to real APIs. UI-only: 2FA toggle, sessions list (hardcoded "Active now"), delete-account (no handler), conversations, notifications. The readme's "40%" understates wiring but fairly reflects feature completeness.

**Phase 2 — scale & maintainability**

17. Redis cache for plans / platform config / brand config
